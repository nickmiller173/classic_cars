# Cars & Bids Price Intelligence

An independent machine learning platform built entirely on Cars & Bids auction data. The platform combines a predictive pricing engine with deep market analysis across ~31,000 auctions and ~$883M in recorded sales, offering buyers, sellers, and enthusiasts an objective lens on what cars are actually worth.

The app is live at Streamlit Community Cloud and consists of four tools: a real-time price predictor backed by a serverless AWS Lambda API, a market trends dashboard, an NLP insights explorer, and a model transparency page showing how the predictor works under the hood.

## Project Goals

- Build a price prediction model trained exclusively on Cars & Bids historical auction results
- Deploy that model as a low-cost, serverless API that any user can call from a browser
- Pair the predictor with exploratory market analysis so users can understand the data behind the estimates
- Keep the infrastructure cost near zero when idle (serverless, no always-on compute)

## File-by-File Reference

### `processing/functions.py`
The shared library of cleaning and feature engineering functions used by `preprocessing.ipynb`. Anything that transforms raw scraped data lives here so the logic is testable and reusable.

**Key functions:**
- `clean_currency`, `clean_mileage` — strip formatting from raw scraped strings
- `clean_and_group_title` — collapses raw title variants into ~6 broad buckets to keep cardinality manageable for encoding
- `extract_engine_info` — parses engine strings like `"4.0L Flat-6"` into separate displacement (float) and cylinder config (string) features; returns a tuple because both carry independent predictive signal
- `get_main_color` — maps exotic color names (e.g. "Estoril Blue", "Bianco Monocerus") to standard color buckets; `special_map` is checked before generic substring matching to prevent collisions like "titanium" matching "tan"
- `engineer_sharp_features` — the master NLP feature builder; creates all boolean flag indicators (one owner, two keys, dry climate car, etc.) and the flaw severity score via regex on listing text fields
- `assign_trim_tier` — maps a URL-derived trim slug to one of 5 price tiers (ultra_premium → economy) using token set matching checked in priority order so a slug like `gt3-turbo` resolves to the highest matching tier
- `engineer_date_features` — extracts auction year/month and derives `model_year` from the "THIS... is a YYYY" pattern in Highlights text (more reliable than the raw Year column from scraping)

**Important:** This file is mirrored in `deployment/utils.py`. Any changes here must be manually synced to the deployment copy, as Lambda cannot import from the processing directory.

---

### `processing/preprocessing.ipynb`
The main data pipeline. Reads from `cars_and_bids_full_history_v3.csv` and produces cleaned, feature-engineered train/test splits ready for modeling.

**Pipeline steps (in order):**
1. Load and deduplicate raw scraped data
2. Clean all raw fields using `functions.py`
3. Fetch S&P 500 closing prices via `yfinance` and merge on auction date (backward fill to handle weekends/holidays)
4. Engineer all features: age, mileage per year, text flags, trim tier, flaw severity, interaction features
5. Save `cleaned_data_no_encoding.csv` — the full dataset before any ML-specific transforms (used by the Home page for accurate aggregate stats)
6. Apply price floor and 99th percentile ceiling to remove data entry errors
7. Log-transform Sold_Price and Mileage (compresses right-tail distributions)
8. Apply TargetEncoder, LabelEncoder, and one-hot encoding
9. Save train/test CSVs and encoding artifacts

**Critical note:** The interaction features (`make_model_mileage_bucket`, `car_age_x_mileage`, etc.) must be created using the **raw mileage value before the log transform**. The log-transformed Mileage column is used only by the model as a direct feature.

**Outputs:** `cleaned_data_no_encoding.csv`, train/test CSVs, `model_artifacts_002.pkl`, `encoding_artifacts_002.pkl`

**Dependencies:** `cars_and_bids_full_history_v3.csv`

### `processing/average_price_options.ipynb`
Generates `historical_averages_lookup.csv` — a flat file of individual auction rows used by the predictor to compute dynamic historical averages. Rows are kept flat (not pre-aggregated) so the predictor can apply a waterfall of filters at runtime (Make+Model+Year+Trans → Make+Model+Year → Make+Model → Make) and always return the tightest available comp set.

**Dependencies:** `cars_and_bids_full_history_v3.csv`
**Output:** `data/frontend_data/historical_averages_lookup.csv`

---

### `NLP/NLP_analysis.ipynb`
Runs all NLP analyses on the auction listing text fields and exports four CSVs consumed by the Text Analysis page.

**Analyses:**
- **General keyword flags** — computed directly from `dashboard_data.csv` (the engineered boolean indicators)
- **Aftermarket brand extraction** — regex-based extraction of premium brand names from Modifications/Equipment text
- **Listing archetypes** — NMF (Non-negative Matrix Factorization) with 4 topics on TF-IDF vectors of combined listing text; NMF was chosen over LDA because it produces parts-based decompositions that are more interpretable on short auction descriptions
- **Listing detail / effort scores** — per-section word counts as a proxy for seller effort
- **Auction buzzwords** — Ridge regression (L2) on TF-IDF features to estimate per-word price correlation; Ridge is preferred over OLS because the ~2,500 TF-IDF features are highly correlated

**Outputs:** `nlp_archetypes.csv`, `nlp_brands.csv`, `nlp_buzzwords.csv`, `nlp_effort_scores.csv`

---

### `training/training.ipynb`
Trains and evaluates the prediction model. Both Random Forest and XGBoost are trained; XGBoost was selected as the production model.

**Key choices:**
- **TimeSeriesSplit** for cross-validation instead of KFold — auctions are sorted chronologically, so standard KFold would leak future data into training folds
- **TargetEncoder** for high-cardinality string features (`make_model_year`, `make_model_mileage_bucket`, `make_model_trim`, `seller_x_title`) — maps each category to its mean log-price in the training fold
- **GridSearchCV** is preserved but pinned to single best-known values so it runs in one pass without re-searching; to actually grid search again, expand the param lists
- **Best XGBoost params:** `n_estimators=500`, `learning_rate=0.05`, `max_depth=8`, `min_child_weight=5`, `gamma=0`, `subsample=0.7`, `colsample_bytree=0.6`, `reg_lambda=1`
- **Log1p price target** — the model predicts log-transformed price; predictions are reversed with `np.expm1` at inference time

**Outputs:** `model_artifacts_002.pkl` (trained pipeline + training columns list), `encoding_artifacts_002.pkl` (TF-IDF vectorizer, SVD model, label encoders)

---

### `training/model_insights.ipynb`
Generates the three CSVs that power the Prediction Analysis page. Run this after retraining.

**Produces:**
- **Residual data** — test set predictions vs. actuals (uses test set only; using training set would show artificially high accuracy due to overfitting)
- **SHAP importance** — mean absolute SHAP value per feature using `TreeExplainer` (exact and fast for XGBoost, unlike the slower model-agnostic `KernelExplainer`); `Year` is back-calculated from `model_year` in the output because it was dropped from features during training to avoid collinearity
- **PDP data** — partial dependence values for each continuous feature across its observed range

**Outputs:** `shap_importance.csv`, `residual_data.csv`, `pdp_data.csv`

---

### `deployment/app.py`
The AWS Lambda handler. This is the entry point called on every prediction request.

**Request flow:**
1. Deserialize the JSON body from API Gateway
2. Short-circuit immediately if `warmup: true` (EventBridge keep-warm ping)
3. Derive age, mileage, and make-model-year features from raw inputs
4. Normalize and combine all listing text fields; create the text blob **before** calling `engineer_sharp_features` because that function internally drops `full_text_blob`
5. Run `engineer_sharp_features` to generate all boolean NLP flags and flaw severity score
6. Derive trim tier and interaction features
7. Fetch live S&P 500 close via `yfinance` (fallback: 5000.0)
8. Apply label encoders for color columns; unseen values map to `'Other'` rather than raising an error
9. One-hot encode categorical columns via `pd.get_dummies` (no fitted encoder needed because column alignment is enforced entirely by `reindex`)
10. Transform text blob through TF-IDF → SVD to produce 20 latent text components
11. `reindex` to match exact training column order; `fill_value=0` is safe for one-hot columns
12. Predict log-price and reverse with `np.expm1`

**Dependencies:** `utils.py`, `model_artifacts_002.pkl`, `encoding_artifacts_002.pkl`

A copy of `processing/functions.py` scoped to the functions Lambda actually needs at inference time. Must be kept in sync with `functions.py` manually — Lambda cannot import from the processing directory.

**Functions present:** `assign_trim_tier`, `extract_trim_slug`, `extract_performance_trim_flag`, `categorize_mods`, `calculate_flaw_severity`, `engineer_sharp_features`

### `deployment/Dockerfile`
Defines the Lambda container image. Uses `public.ecr.aws/lambda/python:3.12` as the base (the official AWS Lambda Python runtime), installs dependencies from `requirements.txt`, copies `app.py`, `utils.py`, and both `.pkl` artifacts, and sets `app.lambda_handler` as the entry point.

**Why a container instead of a zip deployment?** The `.pkl` model artifacts plus scikit-learn and XGBoost exceed Lambda's 250MB unzipped package limit. Containers support up to 10GB.

### `frontend/0_Home.py`
The app entry point and landing page. Streamlit uses the root-level file as the default page.

**Two separate data loaders:**
- `load_data()` — prefers `cleaned_data_no_encoding.csv` (pre-price-floor, full dataset) for hero stats like GMV and median price; falls back to `dashboard_data.csv` in environments where the large file isn't present
- `load_dashboard_data()` — always loads `dashboard_data.csv` specifically for the market signals cards, so those numbers match the Text Analysis page (same dataset, same mean-based calculation)

**Market signals** use `mean` (not `median`) to stay consistent with the Text Analysis page, which also uses mean for its premium calculations.

---

### `frontend/pages/1_Price_Predictor.py`
The price predictor UI. Sends a JSON payload to the Lambda API and displays the ML estimate alongside a historical comp lookup.

**Session state:** Cascading dropdowns (Make → Model → Year → Trim → specs) are managed via `st.session_state` manually rather than using Streamlit's `key=` parameter. This is necessary because Streamlit raises a `ValueError` when a previously stored value is no longer in the updated options list — which happens every time a parent dropdown changes.

**Historical comp lookup:** A waterfall of filters from most-specific (Make + Model + Year + Transmission) to least-specific (Make only), always using the tightest match with at least one result.

**Cold start retry:** If the Lambda returns a 503, the predictor automatically retries once with a toast notification. The `timeout=90` on the requests call is set because Lambda cold starts with large pickle artifacts can take 30–60 seconds, and API Gateway's hard 29-second integration timeout may still cut this off — see the [Quirks](#quirks-and-design-decisions) section.

### `frontend/pages/2_Market_Trends.py`
Seven-tab exploratory dashboard covering platform-wide price trends, per-make/model trends, sales volume, a seasonal heatmap, the model year sweet spot tool, depreciation curves, and price distribution by body style.

**Tabs:**
1. **Market Overview** — monthly average price, top/bottom 10 makes by avg price, private party vs. dealer breakdown
2. **Price by Make & Model** — per-make/model price trend over time with an optional model year filter to isolate a specific vintage
3. **Sales Volume** — monthly listing count across the full platform
4. **Seasonal Trends** — price heatmap by month and year
5. **Model Year Sweet Spot** — bar chart of avg price by production year for a selected make/model
6. **Depreciation by Make** — multi-line chart of avg sale price vs. vehicle age at auction for the top 8 makes; reveals which brands appreciate vs. depreciate
7. **Price by Body Style** — box plot of sale price distribution for each body style (Coupe, Sedan, SUV/Crossover, etc.)

**Filtering thresholds:**
- Price trend tab: only shows make/model pairs with ≥3 months having ≥2 sales, and filters out individual months with only 1 sale (a single unusual auction makes its price identical to the monthly average, creating a misleading spike)
- Sweet spot tab: requires ≥2 qualifying model years with ≥3 sales each (a one-bar chart has nothing to compare)
- Depreciation tab: requires ≥3 sales per make/age bucket; vehicle age capped at 50 years

**Data sources:** `dashboard_data.csv` (all tabs except seller type and body style), `cars_and_bids_full_history_v2.csv` (seller type breakdown and body style box plot — the only raw file that carries those columns)

### `frontend/pages/3_Text_Analysis.py`
Five-tab NLP analysis page. Each tab is powered by a separate pre-computed CSV from `NLP_analysis.ipynb`.

**Important caveat baked into the UI:** The premium percentages shown (e.g. "+19% for two keys") are correlations, not causes. Cars described as project cars sell for less because they're project cars, not because of the words used.

**Data sources:** `dashboard_data.csv` (Tab 1), `nlp_brands.csv`, `nlp_archetypes.csv`, `nlp_effort_scores.csv`, `nlp_buzzwords.csv`

---

### `frontend/pages/4_Prediction_Analysis.py`
Model transparency page with three charts: residual scatter, SHAP feature importance, and partial dependence plots.

- **Residual scatter:** Points above the diagonal sold for more than the model expected; points below sold for less. Color intensity reflects error magnitude.
- **SHAP importance:** Bar length = mean absolute SHAP value = average impact magnitude regardless of direction. A feature that alternately adds and subtracts $5K still scores high.
- **PDP:** Shows the marginal effect of one variable in isolation (holding all others at their mean). Distinct from SHAP, which captures each feature's actual contribution per prediction including interactions.

**Data sources:** `shap_importance.csv`, `residual_data.csv`, `pdp_data.csv`

Defines the custom warm beige color theme applied globally across all pages. Primary color `#8B5E3C`, background `#FAF8F4`, secondary background `#EDE8DF`.

## AWS Infrastructure

### AWS Lambda
The serverless compute layer that runs the prediction model. Lambda is billed per invocation and per GB-second of compute — with typical usage, monthly cost is effectively zero.

**Configuration:**
- **Runtime:** Python 3.12 (via container image)
- **Memory:** 512MB (minimum needed to load scikit-learn + XGBoost artifacts)
- **Timeout:** 90 seconds (cold start + pickle deserialization takes 20–60s)
- **Trigger:** API Gateway (HTTP POST)

**Why Lambda instead of a persistent server?** A traditional server running 24/7 to serve occasional prediction requests would cost $20–50/month. Lambda costs fractions of a cent per request and scales to zero between uses.

### AWS ECR (Elastic Container Registry)
Stores the Docker container image. Lambda pulls the image from ECR on cold start. The repository is private.

**Why a container?** Lambda's zip deployment limit (250MB unzipped) is too small for the model artifacts plus scikit-learn and XGBoost. Containers support up to 10GB.

### AWS API Gateway
The public HTTPS interface in front of Lambda. Handles authentication, request routing, and HTTPS termination.

**Hard timeout limit:** API Gateway enforces a maximum integration timeout of **29 seconds**, which cannot be increased. This is shorter than Lambda's 90-second timeout, meaning a cold start that takes longer than 29 seconds will return a 503/504 before Lambda finishes loading. This is mitigated by the EventBridge warm-up schedule.

### AWS EventBridge (Scheduler)
A scheduled rule fires every 10 minutes sending a warmup ping (`{"warmup": true}`) to the Lambda. The Lambda handler short-circuits immediately on warmup requests, keeping the container alive without running inference. This prevents users from hitting cold starts.

**Cost:** ~4,320 invocations/month, well within the free tier.

## Streamlit Frontend

### Streamlit Community Cloud
The frontend is deployed on Streamlit Community Cloud, which hosts Streamlit apps directly from a GitHub repository at no cost for public projects.

**Entry point:** `frontend/0_Home.py` (configured in Streamlit Cloud deployment settings as the main file)

**Page routing:** Streamlit's multipage app convention uses the root-level file as the home page and discovers additional pages from the `pages/` subdirectory. File names use a `N_PageName.py` convention — the leading number controls sidebar order and Streamlit strips it from the URL slug (e.g. `pages/1_Price_Predictor.py` → `/Price_Predictor`).

**Why Streamlit?** Streamlit lets data scientists build interactive web apps entirely in Python without any frontend framework. The tradeoff is limited layout control and no true client-side state, which is why session state management for cascading dropdowns requires manual handling.

### Altair
All charts are built with Altair (Vega-Lite under the hood). Altair was chosen over Matplotlib/Seaborn because it produces interactive, browser-native charts with tooltips, and integrates cleanly with Streamlit via `st.altair_chart`.

## The ML Pipeline

### Feature Engineering Summary
| Feature | Type | Notes |
|---|---|---|
| `car_age` | Numeric | Years since model year at time of auction |
| `mileage_per_year` | Numeric | Raw mileage / (car_age + 0.5); +0.5 prevents division by zero for new cars |
| `model_year` | Numeric | Extracted from Highlights text ("THIS... is a YYYY"); more reliable than raw Year column |
| `make_model_year` | Categorical (TargetEncoded) | Generation-level pricing signal |
| `trim_tier` | Categorical (OHE) | 5 tiers from ultra_premium to economy, derived from URL slug |
| `flaw_severity_score` | Numeric | Additive penalty from Known Flaws + Seller Notes; higher = worse |
| `text_component_0..19` | Numeric | 20 SVD components from TF-IDF on combined listing text |
| `make_model_mileage_bucket` | Categorical (TargetEncoded) | Groups same model at different wear levels |
| `make_model_trim` | Categorical (TargetEncoded) | Model × year × trim pricing signal |
| `seller_x_title` | Categorical (TargetEncoded) | Interaction between seller type and title status |
| `car_age_x_mileage` | Numeric | Joint deterioration signal |
| `sp500_x_auction_year` | Numeric | Market conditions proxy |
| `dry_climate_x_car_age` | Numeric | Climate preservation signal amplified by age |
| `flaw_severity_x_model_year` | Numeric | Older cars penalized more heavily for flaws |

### Text Features
Listing text fields (Highlights, Equipment, Modifications, Known Flaws, Recent Service History, Ownership History, Seller Notes, Other Items Included in Sale) are processed two ways:

1. **Regex boolean flags** — specific keyword patterns detect conditions like single ownership, two keys, dry climate origin, project car status, etc. These only trigger on exact phrases, not casual synonyms.
2. **TF-IDF + SVD** — the combined text blob is vectorized with a fitted TF-IDF vectorizer (trained on the full corpus) and reduced to 20 dense latent components via Truncated SVD. These capture distributional text patterns that the regex flags miss but respond to full listing text, not casual user-typed phrases.

### Why XGBoost Over Random Forest?
Both were trained and evaluated. XGBoost produced lower RMSE and MAPE on the held-out test set, particularly for high-value vehicles where tree depth and gradient boosting's sequential error correction outperformed the RF's independent tree averaging.

### Price Transformation
Sale price is log-transformed (`log1p`) before training and predictions are reversed with `np.expm1` at inference. This compresses the right-tail distribution of prices, gives the model uniform percentage error sensitivity across price segments (a $2,000 error on a $10,000 car matters more than on a $200,000 car), and stabilizes gradient updates during boosting.

## Deployment Workflow

When the model is retrained, the following sequence is required to push the update to production:

1. **Retrain** — run `training/training.ipynb`; new `.pkl` files are saved
2. **Update model insights** — run `training/model_insights.ipynb` to regenerate SHAP, PDP, and residual CSVs
3. **Regenerate frontend data** — run `processing/dashboard_data.ipynb`, `dropdown_options.ipynb`, `average_price_options.ipynb` if preprocessing also changed
4. **Copy artifacts** — move `model_artifacts_002.pkl` and `encoding_artifacts_002.pkl` to `deployment/`
5. **Build Docker image** — from `deployment/`: `docker build -t classic_car_predictor .`
6. **Push to ECR** — tag and push the image to the AWS ECR repository
7. **Deploy to Lambda** — in the Lambda console, update the container image URI to the new ECR image
8. **Push frontend changes to GitHub** — Streamlit Community Cloud auto-redeploys on push to main

## Quirks and Design Decisions

**API Gateway 29-second hard limit**
Lambda is configured with a 90-second timeout, but API Gateway caps integration responses at 29 seconds regardless. Cold starts (container spin-up + pickle deserialization) can exceed this, causing a 503. The EventBridge 10-minute warm-up ping mitigates this in practice. If cold starts become a persistent problem, switching from API Gateway to a Lambda Function URL removes the 29-second cap.

**Two separate data loaders on the Home page**
The Home page hero stats (GMV, median price, auction count) use `cleaned_data_no_encoding.csv` — the full pre-price-floor dataset. The market signals cards (two keys premium, emissions premium) use `dashboard_data.csv` to stay consistent with the Text Analysis page, which also uses mean-based calculations on the same file. Using two different datasets for two different purposes on the same page was intentional.

**`cleaned_data_no_encoding.csv` is gitignored**
At 194MB it exceeds GitHub's file size limits. It is regenerated by running `preprocessing.ipynb`. The app falls back to `dashboard_data.csv` if it's not present, so the deployed app always works — the Home page hero stats will just reflect the slightly smaller post-processing dataset.

**Model artifacts are gitignored**
`.pkl` files are gitignored. They live only in `deployment/` locally and inside the Lambda container image in production. They are never committed to the repo.

**`utils.py` is a manual mirror of `functions.py`**
Lambda runs in an isolated container and cannot import from the `processing/` directory. Any changes to feature engineering logic in `functions.py` must be manually duplicated in `deployment/utils.py` and a new container image must be built and deployed for the change to take effect at inference time.

**Text fields in the predictor respond to specific keywords, not free text**
The model was trained on full, detailed listing text from Cars & Bids. Typing casual phrases into the predictor text fields has little effect because (1) regex flags only trigger on exact keyword patterns and (2) the TF-IDF/SVD components are latent topics calibrated to the full corpus — a few words don't move them meaningfully. For the most accurate prediction, paste actual listing text from a live Cars & Bids auction.

**Cascading dropdowns require manual session state**
Streamlit's native `key=` parameter for widgets stores the widget value in session state automatically, but raises a `ValueError` when the stored value is no longer in the new options list. Since each dropdown filters the next (Make → Model → Year → Trim → specs), this happens on every parent change. All cascading dropdowns in the predictor manage session state manually to avoid this.

**`numpy<2` pin in deployment requirements**
Numpy 2.x introduced breaking changes in array serialization. Model artifacts pickled with scikit-learn on numpy 1.x cannot be deserialized on numpy 2.x without errors. The pin ensures the Lambda container uses a compatible numpy version.

**Shap uses TreeExplainer, not KernelExplainer**
`TreeExplainer` is exact and fast for tree-based models like XGBoost. `KernelExplainer` is model-agnostic but requires sampling and is orders of magnitude slower on large datasets. The pipeline must be unpacked before calling SHAP because SHAP needs direct access to the XGBoost model object, not the sklearn Pipeline wrapper.

**`shap_importance.csv` values are in log-dollar space**
SHAP values are computed on the model's log-transformed predictions, so the units are log($). This is why the SHAP chart x-axis is labeled "Mean Absolute SHAP Value (log $)" — a SHAP value of 0.5 does not mean $0.50 of impact; it means a 0.5-unit shift in log-price, which translates to roughly a 65% multiplicative effect on the predicted price.
